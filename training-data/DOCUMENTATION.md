# Training-Data Pipeline Documentation

This directory contains a self-contained machine learning mini-project that counts
exercise repetitions from wearable IMU sensor data. The pipeline ingests raw motion
recordings, trains a classifier, and exports it as JavaScript for on-device inference
inside the app—no server required at prediction time.

## 1. Project Overview

The goal is to automatically count gym exercise repetitions (reps) from IMU sensor
data collected by a wearable device (specifically a WT901 BLE sensor). Rather than
predicting a total rep count from a whole recording in one shot, this pipeline uses a
**Segment-and-Score** architecture:

1. Pick the single best sensor axis for this recording.
2. Slice the signal into many *candidate* rep-shaped segments (deliberately more than the actual rep count).
3. Classify each candidate as **real rep** or **noise** using a trained Random Forest.
4. Count the survivors.
5. For each surviving rep, analytically detect **Phase A** (lifting direction) and **Phase B** (return direction) and measure speed.

This approach is robust because the classifier operates on local segment features
rather than global recording statistics, generalises across exercise types through
categorical one-hot features, and requires only a single label per recording (total
rep count)—not per-rep boundary annotations.

## 3. Data Format

Every recording is a JSON file with the following top-level fields:

| Field           | Type              | Description |
|-----------------|-------------------|-------------|
| `reps`          | `int`             | **Required for training.** Total rep count for this set. |
| `samples`       | `array`           | Time-series sensor samples (see below). |
| `repMarkers`    | `array` (optional)| Manual per-rep boundary annotations (see [Labeling Workflow](#7-labeling-workflow)). |
| `muscleGroup`   | `string`          | Primary muscle group (e.g. `"chest"`, `"legs"`). |
| `equipmentType` | `string`          | Equipment used (e.g. `"barbell"`, `"dumbbell"`). |
| `mechanicType`  | `string`          | Movement pattern (e.g. `"compound"`, `"isolation"`). |
| `setNumber`     | `int`             | Which set this is within the workout session. |
| `exerciseName`  | `string` (optional)| Human-readable exercise name. |
| `startedAt`     | `string` (optional)| ISO 8601 wall-clock timestamp when recording began on device. Used to align chart with video. |

Each element of `samples` has this structure:

```json
{
  "timestamp": 1718000000000,
  "angle": { "x": 12.3, "y": -5.1, "z": 0.8 },
  "accel": { "x": 0.02, "y": 0.01, "z": 1.00 },
  "gyro":  { "x": 1.5,  "y": -0.3, "z": 0.1  }
}
```

- `timestamp`: Unix epoch milliseconds (monotonic from the device).
- `angle` (x/y/z): Euler angles in degrees. Computed on the WT901 sensor itself from its onboard Mahony/Madgwick filter.
- `accel` (x/y/z): Linear acceleration in units of **g** (1 g ≈ 9.807 m/s²).
- `gyro` (x/y/z): Angular velocity in **degrees per second**.

Each element of `repMarkers` (if present) has:

```json
{ "startMs": 1718000001200, "endMs": 1718000003800 }
```

Both values are absolute epoch milliseconds matching the `timestamp` field of `samples`.

## 4. Script Reference

### 4.1 `train.py`

**Purpose:** Build and export the rep-segment classifier from all recordings in `raw-data/`.

**Usage:**
```bash
cd training-data
python train.py
```

**Inputs:** `raw-data/*.json` — every file with a `reps` field is used.

**Outputs:**
- `output/features.csv` — Full per-segment feature matrix (useful for manual inspection and debugging).
- `output/model.pkl` — Python pickle of `{"model": clf, "feature_cols": [...]}`.
- `output/model.js` — Auto-generated JavaScript function (via `m2cgen`) for app-side inference.
- `output/summary.txt` — Leave-one-recording-out CV metrics and feature order.
- `output/sus_data.txt` — Flagged recordings (noisy pseudo-labels, under-segmentation, large LOOCV errors).

**Key constants:**

| Constant | Default | Meaning |
|---|---|---|
| `OVER_SEG_PROMINENCE_FRAC` | `0.03` | Minimum peak/valley prominence as a fraction of signal range. Lower = more segments detected. |
| `MIN_SEG_DURATION_MS` | `300` | Discard segments shorter than 300 ms (sub-rep micro-movements). |
| `MIN_HALF_REP_MS` | `150` | Minimum time between adjacent peaks/valleys, used to set `min_distance` in `find_peaks`. |
| `_BP_LO_HZ` | `0.10` | Bandpass lower bound (corresponds to 10 s/rep — slowest possible controlled movement). |
| `_BP_HI_HZ` | `3.00` | Bandpass upper bound (corresponds to 333 ms/rep — fastest loaded movement). |

**Categorical feature lists (`MUSCLE_GROUPS`, `EQUIPMENT_TYPES`, `MECHANIC_TYPES`):**

These lists are fixed and sorted. Their order determines the positions of the one-hot
encoded columns in the feature vector. **Do not reorder or add to these lists without
retraining the model**, because the feature vector size must remain constant between
training and inference.

**Pipeline executed in `main()`:**

1. `build_segment_dataset()` — iterates all recordings, returns a flat DataFrame.
2. `loocv_by_recording()` — leave-one-recording-out cross-validation.
3. Prints recording-level predictions, MAE, exact-match rate.
4. Writes `sus_data.txt` (noisy ratio > 3.5×, sparse ratio < 1.3×, LOOCV error ≥ 3).
5. Trains a final classifier on **all** data.
6. Exports `model.pkl` and `model.js`.
7. Writes `summary.txt`.

### 4.3 `visualize_loocv.py`

**Purpose:** For every training recording, train a LOOCV model (excluding that recording),
run the full prediction pipeline on it, and generate a standalone interactive HTML
report showing exactly what the model sees and how it decides.

**Usage:**
```bash
cd training-data
python visualize_loocv.py
```

**Outputs:**
- `output/html/<recording_name>.html` — one detailed report per recording.
- `output/html/index.html` — summary table linking all reports, sorted by absolute error.

**What each HTML report shows:**

1. **Metadata badges** — muscle group, equipment, mechanic, set number, sample count, duration, candidate count, compressed payload size.
2. **Stats panel** — ground truth reps, predicted reps, error (✓ or ±N in red/green), average rep duration, total time under tension (TUT).
3. **Side-by-side signal charts:**
   - Left: Full signal at up to 1000 points.
   - Right: Compressed signal (10 pts/rep, minimum 50 pts) with a Download JSON button.
   - Both share the same y-axis scale. Coloured background bands show candidate segments:
     - **Green** = true positive (pseudo-labeled rep + predicted rep)
     - **Orange** = false negative (pseudo-labeled rep, not predicted)
     - **Red** = false positive (predicted rep, pseudo-labeled noise)
     - **Grey** = true negative (both noise)
   - **Yellow dashed vertical lines** mark turning points of predicted reps.
4. **Reconstructed signal** — draws Phase A and Phase B of each predicted rep as straight-line segments (3 key points: start, turning, end), on the same y-axis scale for direct comparison with the raw signal above.
5. **Phase Duration bar chart** — stacked bar per rep showing Phase A and Phase B duration.
6. **Phase Speed line chart** — Phase A and Phase B speed in deg/s per rep.
7. **Per-rep breakdown table** — total duration, Phase A/B durations, speeds, classifier confidence; plus an averages row.

**`train_excluding(df, held_out_rec)`:** Trains a fresh `RandomForestClassifier` on
all rows from `df` except those with `recording_id == held_out_rec`. This is the
exact LOOCV model used to generate predictions for the per-recording HTML page.

**`build_compressed_chart_payload(...)`:** Produces a compact dict that represents
what would actually be stored in the app database alongside the rep count:
- Trims the signal to the window spanning the first predicted rep start to the last predicted rep end (discards setup/unrack tails).
- Downsamples to `max(n_reps * 10, 50)` points.
- Includes segment colour annotations, turning-point timestamps, and Phase A/B reconstructed points (3 key points each, with `null` breaks between reps to prevent line-connecting across gaps).
- Includes y-axis bounds.
- Includes provenance metadata (original sample count, trimmed sample count, downsampled point count).

### 4.4 `generate-markers-html.py`

**Purpose:** Generate a chart-only interactive HTML annotation tool for each recording
in `raw-data/`. No video — useful when only sensor data is available.

**Usage:**
```bash
cd training-data
python generate-markers-html.py              # all recordings
python generate-markers-html.py bench.json  # specific file(s)
```

**Outputs:**
- `output/markers/<recording_name>.html` — one annotation page per recording.
- `output/markers/index.html` — overview table with annotation status badges.

**How annotation works in the browser:**

1. The page loads a Plotly chart showing 12 channels (3 dead-reckoned position axes visible by default, 9 raw sensor axes toggleable via legend).
2. Click **Mark rep** to enter marking mode.
3. Click the chart at the **start** of a rep — a yellow dotted line appears.
4. Click the chart at the **end** of the rep — a blue band is drawn and the marker is saved.
5. Repeat for each rep. Undo and Clear buttons are available.
6. **Download repMarkers.json** — exports only the `[{startMs, endMs}, ...]` array.
7. **Download Full updated JSON** — exports the complete original JSON with `repMarkers` merged in. Replace the original file in `raw-data/` and retrain to use the markers.

**Dead-reckoning in the chart:** The default visible channels (`pos.x`, `pos.y`,
`pos.z` in metres) are computed by `ble_dead_reckoning.compute_position()`. These
world-frame position traces often show rep structure more clearly than raw accel/gyro
because they separate the "up/down" motion from gravity and orientation effects.
The raw axes are always available via the Plotly legend.

**Downsampling:** The chart uses at most 4000 points regardless of recording length,
using uniform index selection (`np.linspace`). The dead-reckoning is run on the
**full** sample array first (integration requires full sample rate), then the result
is downsampled along with the raw channels.

**Index badge logic:**
- **Green `N markers ✓`** — marker count ≥ expected rep count.
- **Orange `N / M markers`** — partially annotated.
- **Orange `no markers`** — not yet annotated.

**How annotation works in the browser:**

- The chart's x-axis uses **relative time from `startedAt`** (the wall-clock moment `handleStart` was called on the phone) as t=0, which is also when the video starts recording. This synchronisation means video time and chart x-coordinate refer to the same physical moment.
- An **orange vertical line** on the chart tracks video playback in real time (~20 fps throttle via `timeupdate`).

**Index page:** `recordings/index.html` shows all subfolders with the same
badge-based annotation status as the other tool. Clicking a row opens that
subfolder's `index.html`.

### 4.6 `ble_dead_reckoning.py`

**Purpose:** Estimate the 3D world-frame position (in metres) of the sensor over time
by integrating the IMU data. Used by both annotation tools to show a more
interpretable signal than raw accel/gyro.

**Usage:** Imported as a module — not run directly.

```python
from ble_dead_reckoning import compute_position
result = compute_position(samples_sorted)
position_m = result["position_m"]  # shape (N, 3)
```

**Algorithm overview (6 steps):**

#### Step 1 — Orientation: Madgwick Filter
`ahrs.filters.Madgwick` fuses raw accelerometer (m/s²) and gyroscope (rad/s) data
to produce a quaternion `q_wxyz` of shape `(N, 4)` at every timestep. The Madgwick
filter is a computationally efficient gradient-descent-based orientation estimator
published in 2010, widely used in embedded IMU applications.

The filter runs at the **estimated sample rate** (`sr_hz = (N-1) / total_duration_s`).

#### Step 2 — Body → World Frame Rotation
The quaternion from step 1 uses `[w, x, y, z]` (Hamilton) convention. scipy's
`Rotation.from_quat` uses `[x, y, z, w]` convention, so the array is rolled by -1
position before constructing the rotation. `rotations.apply(accel_g)` transforms
each raw accelerometer reading from the sensor's local body frame into the fixed
world frame.

After rotation, the nominal gravity vector `[0, 0, 1]` g is subtracted to isolate
linear acceleration. Any remaining DC offset is the bias that step 4 will remove.

#### Step 3 — ZUPT Detection (Zero-Velocity Update)
ZUPT (Zero-Velocity Update) is a classical inertial navigation technique: during
moments when the sensor is stationary, velocity must be zero by definition, so any
accumulated velocity error can be reset.

Stationary windows are detected using a **rolling window** (length `zupt_window_s *
sr_hz`, forced odd) of:
- **Accelerometer magnitude standard deviation** — below `zupt_accel_std_g` (default 0.025 g).
- **Gyroscope magnitude rolling mean** — below `zupt_gyro_dps` (default 8 deg/s).

A sample is marked stationary if **both** conditions hold. The `_runs_of_true()`
helper finds contiguous runs of stationary samples; only runs of at least
`min_stationary_samples` (default 5) samples are kept to avoid false triggering on
transient noise.

#### Step 4 — Bias Interpolation
For each stationary run, the mean linear acceleration is computed as the **bias
estimate** for that window. These anchor estimates are then linearly interpolated
across the full recording using `np.interp`, producing a per-sample bias vector.
When no stationary windows are found (e.g., constant-motion recordings), the global
mean is used as a flat bias.

The debiased linear acceleration in m/s² is:
```
linear_acc_ms2 = (accel_world_g - [0,0,1] - accel_bias_g) * G_MS2
```

#### Step 5 — Velocity by Trapezoidal Integration + ZUPT Reset
`scipy.integrate.cumulative_trapezoid` integrates `linear_acc_ms2` over time `t_s`
to produce velocity. After integration, velocity is **hard-zeroed** at all stationary
samples.

A **linear-ramp drift correction** is applied between each pair of adjacent
stationary windows: the residual velocity at the start of the next stationary window
(which should be zero but isn't due to integration drift) is spread linearly across
the motion segment between them. This is equivalent to assuming constant drift rate
across each swing, which is the standard "Foxlin correction" used in pedestrian
dead-reckoning.

#### Step 6 — Position by Trapezoidal Integration
`scipy.integrate.cumulative_trapezoid` integrates the corrected velocity over time
to produce position in metres. Position starts at `[0, 0, 0]` (the sensor's initial
location).

**Return dict keys:**

| Key | Shape | Description |
|-----|-------|-------------|
| `timestamps_ms` | `(N,)` | Original timestamps |
| `position_m` | `(N, 3)` | World-frame position in metres |
| `velocity_ms` | `(N, 3)` | World-frame velocity in m/s |
| `linear_acc_ms2` | `(N, 3)` | Debiased world-frame linear acceleration in m/s² |
| `accel_bias_g` | `(N, 3)` | Per-sample estimated accelerometer bias in g |
| `stationary` | `(N,)` bool | True where the sensor was detected as stationary |
| `sr_hz` | scalar | Estimated sample rate |
| `orientation_q` | `(N, 4)` | Quaternion `[w, x, y, z]` at every sample |

## 5. Algorithm Deep-Dive

### Step 1 — Best-Axis Selection & Preprocessing

**Function:** `preprocess_to_1d(samples)` in `train.py`

The sensor provides 10 candidate signals:
- 3 Euler angles (x/y/z) — unwrapped to handle ±180° discontinuities
- 3 accelerometer components (x/y/z)
- 3 gyroscope components (x/y/z)
- 1 accelerometer magnitude √(ax² + ay² + az²) — orientation-agnostic

For **each** of the 10 candidates:
1. **Unwrap** (Euler angles only): `np.degrees(np.unwrap(np.radians(ang), axis=0))` removes ±180° jumps that would appear as large spikes in exercises with wide range of motion (e.g., full hip hinge, bicep curl).
2. **Savitzky-Golay smooth:** Window = `max(5, ceil(0.5% of samples) | 1)` (forced odd), polynomial order 2. This suppresses high-frequency noise while preserving peak/valley shape better than a simple moving average.
3. **Butterworth bandpass 0.1–3 Hz:** 2nd-order zero-phase filter (`filtfilt`) that removes slow drift (below 0.1 Hz, e.g., sensor tilting over time) and high-frequency vibration (above 3 Hz). The 0.1–3 Hz band spans all possible human rep speeds from extremely slow (6 reps/min = 0.1 Hz) to maximum speed unloaded plyometrics (180 reps/min = 3 Hz).
4. **Spectral score:** `peak_power / total_power` in the 0.1–3 Hz band. A clean, rhythmic rep signal has most of its energy concentrated at one dominant frequency (the rep rate), giving a high score. A noisy or flat signal has broadband energy, giving a low score.

The candidate with the **highest spectral score** becomes the 1D canonical signal. In
practice, angle axes win for exercises with clear rotation (squat, bench press,
deadlift) and accel axes win when the device doesn't rotate much (lat pulldown).

**Fallback:** If all 10 candidates have range < 1e-4 (effectively flat), the bandpass
of angle-z is used unconditionally. This produces a near-zero signal that results in
no segments, which is handled gracefully downstream.

### Step 2 — Over-Segmentation

**Function:** `over_segment(signal_1d, timestamps)` in `train.py`

The goal is to find **more** candidate segments than actual reps, so the classifier
has noise segments to reject. This is achieved by using a **low** peak detection
threshold (`prominence ≥ 3% of signal range`).

**Algorithm:**
1. Find all **valleys** (local minima) using `scipy.signal.find_peaks(-signal_1d, ...)`.
2. Find all **peaks** (local maxima) using `scipy.signal.find_peaks(signal_1d, ...)`.
3. `min_distance` for both = `max(1, sample_rate * 150ms / 1000)` — prevents two turning points within 150 ms of each other (enforces minimum half-rep duration).
4. Build **valley-to-valley** segments: consecutive valley pairs define the start/end of each segment; the internal turning point is the maximum between them.
5. Build **peak-to-peak** segments: consecutive peak pairs define segments; internal turning point is the minimum.
6. Discard any segment shorter than 300 ms.
7. Choose whichever set (valley-to-valley or peak-to-peak) has **more segments**. If equal, choose the one with lower duration coefficient of variation (CV = std/mean of segment durations), because more consistent durations suggest better-aligned segment boundaries.

Each segment dict stores:
- `start_idx`, `turning_idx`, `end_idx` — sample array indices
- `start_ts`, `turning_ts`, `end_ts` — timestamps in epoch milliseconds

### Step 3 — Feature Extraction

**Function:** `extract_segment_features(seg_idx, seg, signal_1d, timestamps, all_segs, metadata)` in `train.py`

Computes 11 signal features + one-hot categorical features for a single candidate
segment. The full feature vector has **`len(SEGMENT_FEATURE_COLS)`** dimensions
(11 + 10 muscle + 13 equipment + 8 mechanic = 42 total, plus any additions).

**Signal features:**

| Feature | Formula / Description |
|---|---|
| `amplitude` | `max(chunk) - min(chunk)` — absolute peak-to-trough range |
| `duration_ms` | `timestamps[end] - timestamps[start]` |
| `energy` | `Σ(chunk² * avg_dt_s)` — integral of squared signal, approximates mechanical work |
| `prominence` | scipy `peak_prominences` of the turning point in the full 1D signal context. Real reps have prominent turning points; noise segments have subtle ones |
| `relative_amplitude` | `amplitude / global_range` — normalised within the recording |
| `relative_duration` | `duration_ms / median_segment_duration` — how this segment compares to typical duration |
| `temporal_regularity` | `1 / (1 + CV_of_inter_segment_intervals)` — higher = more rhythmic. Computed from all segments in the recording. Ranges 0–1; 0.5 if fewer than 3 segments |
| `position_frac` | `(start_ts - rec_start) / rec_duration` ∈ [0,1]. Setup and unrack events cluster near 0 and 1; genuine reps cluster in the middle |
| `neighbour_amp_ratio` | `amplitude / mean(amplitude of adjacent segments)`. A rep that is much larger or smaller than its neighbours is suspicious |
| `set_number` | From metadata; sets later in a session tend to have different fatigue patterns |
| `spectral_entropy` | Shannon entropy of the FFT power spectrum of the segment: `-Σ p*log(p)`. Low entropy = most power at one frequency = rep-like. High entropy = broadband = noisy |

**Categorical features (one-hot encoded):**

- `muscle_<group>` for each of the 10 muscle groups.
- `equip_<type>` for each of the 13 equipment types.
- `mechanic_<type>` for each of the 8 mechanic types.

Unknown values (missing or unrecognised strings) map to the `unknown` bucket in each
category. The lists are sorted alphabetically and fixed — adding a new exercise type
requires updating the list and retraining.

---

### Step 4 — Pseudo-Labeling

**Two labeling strategies exist:**

#### Strategy A — Pseudo-labeling (no `repMarkers` in file)

**Function:** `pseudo_label_segments(segments, n_reps)` in `train.py`

Uses only the known total rep count to assign labels. Each segment is scored:
```
score = amplitude × √(energy + 1e-6) × temporal_consistency
```
where `temporal_consistency = 1 / (1 + |duration - median_duration| / median_duration)`.

This rewards segments that are:
- Large (high amplitude)
- Energetic (high energy)
- Typically paced (duration close to the median for this recording)

The top `n_reps` segments by score are labeled `is_rep=1`; the rest are `is_rep=0`.

**Critical constraint:** If the over-segmenter found *fewer* candidates than the
known rep count, pseudo-labeling is impossible (we can't assign `n_reps` labels to
fewer candidates), so the recording is **skipped** entirely and reported in
`sus_data.txt`.

#### Strategy B — IoU marker matching (file has `repMarkers`)

**Function:** `label_segments_from_markers(segments, rep_markers)` in `train.py`

For each manual marker `{startMs, endMs}`, finds the candidate segment with the
highest **Intersection over Union (IoU)** with the marker's time window:
```
IoU = overlap / (segment_duration + marker_duration - overlap)
```
The best-matching segment is labeled `is_rep=1` if `IoU > 0.05`, and each segment
can only be claimed by one marker. Segments not matched to any marker get `is_rep=0`.

**Strategy B is preferred** when available because the labels are directly human-verified
rather than heuristically inferred from signal characteristics.

---

### Step 5 — Training & Export

**Classifier:** `sklearn.ensemble.RandomForestClassifier`
- `n_estimators=200` — 200 trees; enough for stable importance estimates.
- `max_depth=6` — shallow trees reduce overfitting to the small training set.
- `class_weight="balanced"` — compensates for the class imbalance (more noise segments than rep segments in every recording).
- `random_state=42` — deterministic output for reproducibility.

**JavaScript export via `m2cgen`:**
`m2cgen.export_to_javascript(clf, function_name="classifySegment")` transpiles the
entire trained forest into a pure JavaScript function (no external dependencies, no
WASM). The output is minified (all whitespace collapsed) and wrapped with:
```js
// @ts-nocheck
/* eslint-disable */
export { classifySegment };
```
At runtime: `classifySegment(features)[1] > 0.5` → is a real rep.

The input array must have **exactly `len(SEGMENT_FEATURE_COLS)` elements** in the
order defined by `SEGMENT_FEATURE_COLS`. This order is also written into the
header comment of `model.js` for reference.

---

### Inference — Phase Detection

**Function:** `detect_phases(seg, signal_1d, timestamps)` in `predict.py`

Once the classifier identifies a segment as a real rep, the segment is analytically
split at its turning point (already identified during over-segmentation) into:

- **Phase A** (start → turning point): the first direction of motion.
- **Phase B** (turning point → end): the return direction.

For each phase:
- **Duration** = `timestamps[end] - timestamps[start]` in ms.
- **Speed** = `|signal[end] - signal[start]| / (duration_ms / 1000)` in deg/s.

These are unlabeled directionally (Phase A might be concentric or eccentric depending
on the exercise and device orientation). A future calibration step—using a known
reference pose or a dedicated "lifting against gravity" calibration recording—could
determine which phase is concentric for a given exercise+device combination.

## 6. Evaluation Strategy

**Leave-One-Recording-Out Cross-Validation (LOOCV)** is used because:

1. The dataset is small (tens of recordings).
2. Segments within the same recording are highly correlated (same exercise, same person, same sensor placement)—a random 80/20 split would leak information.
3. LOOCV ensures the model is always tested on data from a recording it has never seen.

**Metrics reported:**

*Segment-level* (binary classification):
- **Precision** = TP / (TP + FP) — of segments predicted as reps, how many were actually reps?
- **Recall** = TP / (TP + FN) — of actual rep segments, how many did the model find?
- **F1** = harmonic mean of Precision and Recall.

*Recording-level* (the end goal — rep count accuracy):
- **MAE** (Mean Absolute Error) = mean of |predicted_count - true_count| across recordings.
- **Exact match** = fraction of recordings where predicted count equals true count.

**`sus_data.txt` flags:**
- **High candidate:rep ratio (> 3.5×):** Too many noise candidates relative to reps. Pseudo-labels may be unreliable because the ranking score may not separate real reps clearly enough.
- **Low candidate:rep ratio (< 1.3×):** Barely over-segmented. The model has little room to reject noise; almost every segment must be labeled a rep.
- **Large LOOCV error (≥ 3):** The held-out model was badly wrong for this recording — worth investigating whether the recording has unusual characteristics.
- **Under-segmented (skipped):** The over-segmenter found fewer candidates than the known rep count, making pseudo-labeling impossible.

## 8. Compressed Chart Payload

`build_compressed_chart_payload()` in `visualize_loocv.py` defines the format of
the compact signal representation that would be stored alongside rep count data in
the app database. Its structure is:

```json
{
  "signal": {
    "x": [0.0, 0.1, 0.2, ...],   // time in seconds (relative to trimmed window start)
    "y": [1.23, 1.45, ...]        // downsampled 1D canonical signal values
  },
  "segments": [
    { "xs": 0.0, "xe": 1.8, "color": "rgba(34,197,94,0.22)" },
    ...
  ],
  "turnings": [0.9, 2.7, ...],    // turning point times in seconds (predicted reps only)
  "phaseA": [
    { "x": 0.0, "y": 1.1 }, { "x": 0.9, "y": 2.3 }, { "x": 0.9, "y": null },
    ...
  ],
  "phaseB": [
    { "x": 0.9, "y": 2.3 }, { "x": 1.8, "y": 1.2 }, { "x": 1.8, "y": null },
    ...
  ],
  "yBounds": { "min": -2.1, "max": 3.4 },
  "signalPoints": 80,
  "trimmedSamples": 620,
  "origSamples": 847
}
```

Key points:
- The signal is **trimmed** to the span `[first_predicted_rep_start, last_predicted_rep_end]` before downsampling, so setup/unrack tails don't waste space.
- `phaseA` and `phaseB` use **`null` y-values** as line-break markers between reps, so `spanGaps: false` in Chart.js draws each rep as a disconnected segment.
- `color` values are one of four fixed RGBA strings corresponding to TP/FP/FN/TN (see [Section 4.3](#43-visualize_loocvpy)).
- The Download JSON button in the HTML pages lets you inspect this payload directly.

## 9. Key Design Decisions & Tradeoffs

**Why Random Forest instead of a neural network?**
The training dataset is small (typically tens of recordings, hundreds of segments).
A Random Forest with `max_depth=6` provides good regularisation, interpretable feature
importances, and exports cleanly to JavaScript via `m2cgen` without needing ONNX,
TensorFlow.js, or any runtime library.

**Why segment-and-score instead of direct regression?**
Direct regression ("count the reps in this whole recording") cannot localise *where*
reps occur, which is required for phase timing. It also generalises poorly because
recordings vary wildly in length and content. By operating on fixed-size local
segments, the classifier sees the same feature space regardless of total recording
duration or rep count.

**Why pseudo-labels instead of requiring full annotation?**
Annotating exact rep boundaries in every recording is time-consuming. Pseudo-labeling
requires only the total count (which is trivially recorded during a workout) and
produces surprisingly accurate labels because the ranking score (amplitude × energy
× temporal_consistency) naturally selects genuine reps over setup/unrack noise.
Per-rep boundary markers are supported as an optional upgrade path via `repMarkers`.

**Why bandpass 0.1–3 Hz specifically?**
These bounds are physiological constants, not data-derived parameters. 0.1 Hz = 6
reps/minute (the absolute slowest deliberate gym tempo). 3 Hz = 180 reps/minute (the
absolute fastest loaded movement). Using data to tune these bounds would risk
over-fitting to the current exercise mix. Lowering `_BP_LO_HZ` toward 0 Hz would
let slow sensor drift compete in the spectral-score axis selection and incorrectly
select a drift-dominated axis.

**Why is the accelerometer magnitude the 10th candidate instead of being part of the
axis loop?**
The three gyroscope axes and three accelerometer axes are naturally tried as a
3-element block. The magnitude is derived from the three accel components and doesn't
have a paired gyro version, so it's appended as a special case after the main loop.

**Why `class_weight="balanced"` in the classifier?**
In every recording, there are more noise segments than rep segments (by design, since
over-segmentation produces ~1.5–3.5× more candidates than reps). Without balancing,
the forest would learn to predict "noise" for almost everything. Balanced weighting
multiplies each class's loss by `total_samples / (n_classes * class_count)`, giving
equal influence to rep and noise examples during training.
